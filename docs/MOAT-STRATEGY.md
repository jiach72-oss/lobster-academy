# 龙虾学院 · 技术护城河战略

> 太子脑暴 | 2026-03-24 16:18

## 核心判断

**当前状态：UI 产品，无技术壁垒，一周可复制。**

**目标状态：三层护城河，每层让复制成本翻倍。**

---

## 三层护城河

### 第一层：Agent 行为录制 SDK（Blackbox）
**地位：地基。没有它，后面都是空中楼阁。**

**做什么：**
- 像 Google Analytics 嵌入网页一样，一行代码嵌入 Agent
- 拦截每次 LLM 调用：输入 prompt、输出 completion、token 用量、延迟
- 拦截每次工具调用：工具名、入参、返回值、耗时
- 拦截每次推理步骤：chain-of-thought 摘要
- 自动脱敏：密码、密钥、PII 自动替换为 `[REDACTED]`
- Ed25519 签名：每条记录签名，防篡改
- 本地存储优先：数据不出本机，企业安心

**为什么是护城河：**
- 需要深入 Agent 框架内部（OpenClaw plugin、LangChain callback、CrewAI hook）
- 不是前端页面能做的事，是系统级集成
- 谁先集成到主流框架，谁就是事实标准
- 数据格式一旦被下游（保险、审计、采购）采纳，迁移成本极高

**类比：**
- Datadog 的护城河不是仪表盘，是嵌入到每个服务里的 Agent
- Segment 的护城河不是 UI，是那行 `analytics.track()` 被嵌入了十万个项目

### 第二层：对抗性评测引擎
**地位：核心差异化。别人有评测，但我们是"攻击你的 Agent 然后评分"。**

**做什么：**
- 自动构建攻击用例库：
  - 提示注入（直接注入、间接注入、多轮注入）
  - 权限越界（尝试访问未授权工具/数据）
  - 逻辑绕过（让 Agent 违反自己的规则）
  - 资源耗尽（大输入、递归调用、无限循环）
  - 数据泄露（诱导 Agent 输出训练数据/配置信息）
- 每个攻击用例独立评分
- 攻击成功率 = 安全评分的核心依据
- 用例库持续更新（对抗性军备竞赛）

**为什么是护城河：**
- 攻击用例库是积累性资产，越用越强
- 需要深入理解每种 Agent 框架的攻击面
- 大厂做的评测是"你的模型有多聪明"，我们做的是"你的 Agent 有多危险"
- 用例库本身就是数据护城河

### 第三层：信任链基础设施
**地位：终极壁垒。从工具变成基础设施。**

**做什么：**
- W3C Verifiable Credentials 格式证书
- 证书吊销列表（CRL）实时同步
- 第三方时间戳（RFC 3161）+ 区块链锚定
- 保险联动 API：保险公司直接查询证书状态决定费率
- 采购联动 API：企业采购 Agent 前自动校验证书
- 监管报告生成器：一键输出符合 EU AI Act 的合规报告

**为什么是护城河：**
- 一旦保险公司和采购系统集成了我们的 API，切换成本 = 改造整个流程
- 这不是技术壁垒，是生态壁垒
- 类比：DigiCert/Let's Encrypt 的护城河不是签名算法，是所有浏览器都信任它们

## 执行优先级

| 优先级 | 组件 | 时间 | 产出 |
|--------|------|------|------|
| P0 | Blackbox SDK (OpenClaw plugin) | 本周 | 能录制 Agent 行为的 npm 包 |
| P0 | 脱敏引擎 | 本周 | 200+ 种 PII/密钥正则 |
| P1 | 对抗性评测引擎 v1 | 下周 | 10 个攻击用例 + 自动评分 |
| P1 | Ed25519 签名模块 | 下周 | 真正的密码学签名 |
| P2 | 证书链 (VC) | 月内 | W3C VC 格式证书 |
| P2 | 评测用例库扩展 | 持续 | 50+ → 200+ 用例 |

## 技术选型

| 组件 | 语言 | 理由 |
|------|------|------|
| Blackbox SDK | TypeScript | Agent 框架多用 TS，npm 生态 |
| 脱敏引擎 | TypeScript | 与 SDK 一体 |
| 签名模块 | TypeScript (tweetnacl) | Ed25519，轻量，npm 可用 |
| 评测引擎 | Python + TS | Python 做攻击生成，TS 做执行 |
| CLI 工具 | TypeScript | 统一技术栈 |
