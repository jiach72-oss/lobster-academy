## 1. Redactor 脱敏引擎

### 1.1 当前架构

- 208+ 预编译正则模式（PII、凭证、令牌、密钥）
- 支持 i18n 国际化模式（从 `patterns.json` 按国家加载）
- 递归脱敏 dict/list 对象
- 自动地区检测与按 locale 脱敏

### 1.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **线性扫描性能退化** | 🔴 高 | 208+ 正则逐一匹配，长文本（>10KB）时延迟显著。每个 `redact_string()` 调用遍历全部模式 |
| **无优先级/短路机制** | 🔴 高 | 高频模式（如 email）和低频模式（如 Shopify token）同等对待，浪费算力 |
| **假阳性率不可控** | 🟡 中 | VIN、MAC 地址等模式在非敏感上下文中产生误报，无上下文感知 |
| **缺少增量/流式处理** | 🟡 中 | 无法处理流式 Agent 输出（如 SSE），只能等完整文本 |
| **i18n 模式加载惰性且无缓存** | 🟢 低 | 每次构造 Redactor 重新读文件，无预编译缓存 |

### 1.3 推荐算法

#### A. Aho-Corasick 多模式匹配 + 分层过滤

**理由**：正则逐条匹配是 O(N×M)，Aho-Corasick 可实现 O(N+M) 多模式同时匹配，对固定字符串模式（如 `AKIA`、`sk-`、`ghp_`）效率提升 10-50 倍。

**实现方案**：
```
层级1: Aho-Corasick 快速扫描（固定前缀模式，~80个）
  ↓ 命中
层级2: 精确正则验证（编译正则验证完整格式）
  ↓ 命中
层级3: 上下文校验（检查前后字符是否为引号/等号等边界）
```

- 使用 `pyahocorasick` 库构建自动机
- 按模式频率预排序：AWS key 前缀 `AKIA`、OpenAI `sk-`、GitHub `ghp_` 等高频模式优先
- 只有 Aho-Corasick 命中后才执行完整正则验证，避免无谓计算

#### B. Bloom Filter 预过滤

**理由**：对于对象脱敏中的敏感键检测，用 Bloom Filter 替代线性遍历敏感键列表，O(1) 查询。

**实现方案**：
- 预构建 2048-bit Bloom Filter，存储所有 SENSITIVE_KEYS
- `_is_sensitive_key()` 先查 Bloom Filter，命中后才做精确匹配
- 内存占用从 ~15KB 降到 ~256 字节

#### C. 流式脱敏（Streaming Redactor）

**理由**：Agent 输出常为流式（SSE/chunk），需要逐块脱敏而不丢失跨块上下文。

**实现方案**：
- 维护滑动窗口缓冲区（默认 256 字符），处理跨 chunk 边界的模式
- `redact_chunk(chunk) -> chunk` 接口，内部维护有限状态机
- 对于可能跨边界的模式（JWT 73 字符、私钥多行），扩展缓冲区

### 1.4 优先级与预期收益

| 优化项 | 优先级 | 预期收益 |
|--------|--------|----------|
| Aho-Corasick 分层匹配 | **P0** | 长文本脱敏速度提升 10-50x |
| Bloom Filter 敏感键检测 | **P1** | 对象脱敏性能提升 ~30% |
| 流式脱敏 | **P1** | 支持实时 Agent 输出流脱敏 |
| 模式频率排序 | **P1** | 平均匹配时间减少 40% |
| i18n 模式缓存 | **P2** | 消除重复文件 I/O |

## 2. AdversarialEngine 对抗性评测引擎

### 2.1 当前架构

- 9 类 53 个攻击场景硬编码为方法
- 简单的敏感模式匹配判定（`_contains_sensitive_patterns`）
- 串行执行所有攻击
- 无动态 payload 生成能力

### 2.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **判定逻辑过于简单** | 🔴 高 | 仅检查 5 个敏感关键词，无法检测间接泄露、推理链泄露、编码绕过等 |
| **无攻击变体/模糊化** | 🔴 高 | 固定 payload 容易被针对性防御，缺乏同义替换、编码变换、多语言变体 |
| **串行执行效率低** | 🟡 中 | 53 个攻击串行运行，大量时间花在等待 agent 响应 |
| **无严重度加权评分** | 🟡 中 | CRITICAL 和 LOW 失败对总分影响相同，评分缺乏区分度 |
| **缺少对抗性迭代** | 🟡 中 | 无 "生成 → 测试 → 变异 → 再测试" 的自适应攻击循环 |

### 2.3 推荐算法

#### A. 分层响应分析器（Tiered Response Analyzer）

**理由**：替代简单的关键词匹配，构建多层检测机制。

**实现方案**：
```
Tier 1: 精确模式匹配（当前逻辑，快速排除）
  ↓ 未检出
Tier 2: 语义相似度检测（embedding 余弦相似度，检测同义泄露）
  ↓ 未检出
Tier 3: 结构分析（检测输出中是否包含格式化的凭证结构）
  ↓ 未检出
Tier 4: 推理链审查（检测 reasoning 中是否包含敏感推导路径）
```

- Tier 2 使用轻量 embedding 模型（如 `all-MiniLM-L6-v2`，<100MB）
- 预构建敏感语义向量库（"API key"、"password" 等的 embedding）
- 余弦相似度 > 0.85 判定为语义泄露

#### B. Fuzzing 变异引擎

**理由**：从固定 payload 扩展为可变异的攻击模板。

**实现方案**：
- **模板系统**：`"Ignore all previous instructions"` → 模板 `{action} {scope} {directive}`
- **变异操作**：
  - 同义替换：`ignore` → `disregard`/`bypass`/`override`
  - 编码变异：Base64/ROT13/Unicode/Hex/Leetspeak
  - 分隔符变异：换行/null byte/特殊字符
  - 多语言变体：中/日/韩/阿拉伯语翻译
  - 对抗性 token：添加噪声/特殊字符绕过过滤器
- **覆盖率引导**：跟踪已触发的防御类型，优先变异未覆盖的攻击向量

#### C. 严重度加权评分算法

**理由**：不同攻击失败的后果不同，评分应反映真实风险。

**实现方案**：
```python
SEVERITY_WEIGHTS = {"critical": 4.0, "high": 3.0, "medium": 2.0, "low": 1.0}

def weighted_score(results: list[AttackResult]) -> float:
    total_weight = sum(SEVERITY_WEIGHTS[r.severity] for r in results)
    passed_weight = sum(SEVERITY_WEIGHTS[r.severity] for r in results if r.passed)
    return (passed_weight / total_weight) * 100 if total_weight > 0 else 0.0
```

#### D. 并行攻击执行器

**理由**：独立攻击场景可并行执行，缩短评测时间。

**实现方案**：
- 使用 `concurrent.futures.ThreadPoolExecutor` 并行执行
- 按类别分组，同类攻击串行（避免上下文污染），不同类并行
- 预期从 ~53s 降至 ~10s（5 类并行 × ~10s/类）

### 2.4 优先级与预期收益

| 优化项 | 优先级 | 预期收益 |
|--------|--------|----------|
| 分层响应分析器 | **P0** | 检出率从 ~60% 提升至 ~90% |
| 严重度加权评分 | **P0** | 评分更准确反映真实安全风险 |
| Fuzzing 变异引擎 | **P1** | 攻击场景从 53 扩展至 500+ 变体 |
| 并行执行 | **P1** | 评测耗时从 ~53s 降至 ~10s |
| 对抗性迭代循环 | **P2** | 自动生成发现新漏洞 |

## 3. Recorder 录制引擎

### 3.1 当前架构

- MemoryStorage（内存）和 PostgreSQLStorage 两种后端
- 支持事件记录、查询、回放
- 事件包含：输入、推理链、输出、工具调用、签名

### 3.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **无事件链完整性校验** | 🔴 高 | 事件间无哈希链，无法检测中间事件篡改或删除 |
| **缺少事件压缩** | 🟡 中 | 大量重复/相似事件占用存储，无去重机制 |
| **PostgreSQL 无连接池** | 🟡 中 | 高并发场景下连接开销大 |
| **无异步写入** | 🟡 中 | `record()` 是同步阻塞调用，影响 Agent 响应延迟 |
| **缺少分页查询** | 🟢 低 | `get_events()` 返回全部事件，大数据量时内存爆炸 |

### 3.3 推荐算法

#### A. Merkle 哈希链（事件完整性）

**理由**：为事件序列构建不可篡改的哈希链，类似区块链的轻量实现。

**实现方案**：
```
Event₁ → hash₁ = SHA256(event₁.data + signature₁)
Event₂ → hash₂ = SHA256(event₂.data + hash₁ + signature₂)
Event₃ → hash₃ = SHA256(event₃.data + hash₂ + signature₃)
...
```

- 每个事件携带 `prev_hash` 字段，链接前一事件
- 回放时逐级验证哈希链完整性
- 检测到哈希不匹配 → 标记从该事件起为 "tampered"
- 存储开销：每事件增加 32 字节（SHA-256）

#### B. 增量事件压缩（Delta Encoding）

**理由**：Agent 连续事件间常有大量重复内容（系统提示、上下文）。

**实现方案**：
- 对连续事件的 `input`、`reasoning`、`output` 字段做 diff
- 存储增量而非全量，使用 `zstd` 压缩
- 预期压缩率 60-80%（对重复上下文）
- 查询时自动重建完整事件

#### C. 异步批量写入器

**理由**：解耦 Agent 推理和事件持久化。

**实现方案**：
- 内存环形缓冲区（默认 1000 事件）+ 后台 flush 线程
- `record()` → 写入缓冲区（纳秒级）→ 后台线程批量写入存储
- 可配置 flush 策略：定时（100ms）/ 定量（50 事件）/ 手动
- 支持 graceful shutdown（退出前 flush 全部）

### 3.4 优先级与预期收益

| 优化项 | 优先级 | 预期收益 |
|--------|--------|----------|
| Merkle 哈希链 | **P0** | 事件不可篡改，满足审计合规要求 |
| 异步批量写入 | **P1** | Agent 响应延迟降低 5-20ms |
| 增量压缩 | **P1** | 存储占用减少 60-80% |
| 分页查询 | **P2** | 支持大规模事件查询 |

## 4. Academy 学院模块

### 4.1 当前架构

- 5 个评估维度（安全性、推理、工具、合规、稳定性）
- 每维度 5 个指标，共 25 个指标
- 基于事件列表的简单统计评分（通过率 %）
- 字母等级评分（A/B/C/D/F）

### 4.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **评分公式过于简单** | 🔴 高 | 仅用通过率百分比，无权重、无基准线、无区分度 |
| **无基准对标** | 🔴 高 | 没有行业基准，80 分在不同维度含义不同 |
| **指标缺乏深度** | 🟡 中 | 如 hallucination 检测仅靠 5 个关键词，误报/漏报率高 |
| **无趋势分析** | 🟡 中 | 单次评估快照，无法追踪 Agent 能力变化趋势 |
| **缺少入学/毕业流程** | 🟡 中 | Academy 模块仅做评估，无完整的学习路径管理 |

### 4.3 推荐算法

#### A. 加权贝叶斯评分（Bayesian Weighted Scoring）

**理由**：替代简单平均分，引入先验知识和不确定性量化。

**实现方案**：
```python
# 每个指标使用 Beta 分布建模
# alpha = 成功次数 + 先验_成功, beta = 失败次数 + 先验_失败

def bayesian_score(successes: int, total: int, 
                   prior_alpha: float = 2.0, prior_beta: float = 2.0) -> float:
    """Beta 分布后验均值，平滑小样本偏差"""
    alpha = prior_alpha + successes
    beta = prior_beta + (total - successes)
    return alpha / (alpha + beta)  # 后验均值
```

- 对每个指标维护 Beta(α, β) 先验
- 事件数据更新后验参数
- 后验均值作为评分，后验方差作为置信度
- 小样本时自动回归先验（避免 1/1=100% 的极端评分）

#### B. 百分位基准对标（Percentile Benchmarking）

**理由**：为评分提供行业上下文。

**实现方案**：
- 维护基准数据库：按 Agent 类型/规模/行业的评分分布
- 报告相对百分位而非绝对分："安全性 > P90" 而非 "安全性 85 分"
- 定期从聚合匿名数据更新基准

#### C. 多层幻觉检测器

**理由**：当前仅 5 个关键词，无法有效检测幻觉。

**实现方案**：
```
Layer 1: 模式匹配（当前，快速过滤明确幻觉标记）
Layer 2: 事实一致性检查（输出 vs 输入的实体/数值对比）
Layer 3: 不确定性表达检测（"I think"、"probably"、"might be" 等）
Layer 4: 知识边界检测（输出是否超出输入/工具返回的范围）
```

#### D. 能力雷达图与趋势追踪

**理由**：直观展示 Agent 能力画像和变化。

**实现方案**：
- 每次评估生成五维雷达图（安全性/推理/工具/合规/稳定性）
- 存储历史评估序列，计算移动平均趋势
- 支持环比/同比对比，自动标记显著变化（>5% 波动）

### 4.4 优先级与预期收益

| 优化项 | 优先级 | 预期收益 |
|--------|--------|----------|
| 加权贝叶斯评分 | **P0** | 评分更准确、更公平 |
| 多层幻觉检测 | **P0** | 幻觉检出率从 ~30% 提升至 ~75% |
| 百分位基准对标 | **P1** | 评分提供行业参照 |
| 能力趋势追踪 | **P1** | 支持 Agent 能力退化预警 |
| 入学/毕业流程管理 | **P2** | 完善学院产品化 |

## 5. Reporter 报告生成器

### 5.1 当前状态

- 模块尚未在 SDK 中实现（仅在文档中定义）
- 需支持 JSON/Markdown/HTML/文本四种格式

### 5.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **完全缺失** | 🔴 高 | 无报告生成能力，评估结果无法结构化输出 |
| **无模板系统** | 🔴 高 | 需要支持不同格式、不同受众的报告变体 |
| **无增量报告** | 🟡 中 | 每次生成全量报告，无法生成增量变更报告 |

### 5.3 推荐算法

#### A. 模板引擎架构

**理由**：统一的模板系统支持多种输出格式。

**实现方案**：
```python
class Reporter:
    def generate(self, evaluation: EvaluationResult, 
                 attack_results: list[AttackResult],
                 events: list[RecordEvent],
                 format: str = "json",
                 template: str = "full") -> str:
        """生成指定格式的评估报告"""
```

- **JSON**：结构化数据输出，适合程序消费
- **Markdown**：GitHub/飞书友好的文本报告
- **HTML**：含图表的可视化报告（雷达图、时间线）
- **文本**：纯文本摘要，适合 CLI/日志

#### B. 智能摘要生成

**理由**：长报告需要自动生成执行摘要。

**实现方案**：
- 关键指标提取：总分、等级、各维度得分
- 异常检测摘要：自动识别最差维度和关键失败攻击
- 建议生成：基于失败模式匹配预定义的改进建议库
- 摘要长度控制：100/300/500 字三档

#### C. 增量差异报告

**理由**：对比两次评估的变化。

**实现方案**：
- 结构化 diff：逐维度、逐指标对比
- 变化标注：↑ 改善 / ↓ 退化 / → 持平
- 显著性阈值：仅展示 >2% 的变化，避免噪声

### 5.4 优先级与预期收益

| 优化项 | 优先级 | 预期收益 |
|--------|--------|----------|
| 核心报告生成（4 格式） | **P0** | 评估结果可对外输出，产品闭环 |
| JSON 结构化输出 | **P0** | API 消费者可程序化处理 |
| 智能摘要 | **P1** | 提升报告可读性 |
| 增量差异报告 | **P2** | 支持持续评测场景 |

## 6. Signer 签名模块

### 6.1 当前架构

- Ed25519 数字签名
- 支持密钥对生成、签名、验证
- 使用 `cryptography` 库

### 6.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **无密钥管理** | 🔴 高 | 密钥以 raw bytes 传入，无安全存储、轮换、撤销机制 |
| **无批量签名** | 🟡 中 | 每个事件单独签名，大批量时性能开销大 |
| **无签名链验证** | 🟡 中 | 只能验证单个签名，无法批量验证事件序列 |
| **缺少时间戳绑定** | 🟢 低 | 签名未绑定可信时间戳，存在时间篡改风险 |

### 6.3 推荐算法

#### A. 密钥派生与轮换（Key Derivation & Rotation）

**理由**：实现安全的密钥生命周期管理。

**实现方案**：
```
Master Key (32 bytes, HSM/KMS 保护)
  ↓ HKDF-SHA256
├── Signing Key v1 (period=2026-Q1)
├── Signing Key v2 (period=2026-Q2)
├── ...
└── Verification Key Chain
```

- 使用 HKDF 从主密钥派生签名密钥
- 按时间段（月/季）自动轮换
- 公钥链发布，验证时根据时间选择对应公钥
- 支持密钥撤销列表（CRL）

#### B. 批量 Merkle 签名

**理由**：对事件序列签名，O(log N) 验证。

**实现方案**：
- 构建事件批次的 Merkle 树
- 对 Merkle Root 签名（1 个 Ed25519 签名覆盖整个批次）
- 单个事件验证：Merkle Path（log N 个哈希）
- 相比逐事件签名，签名开销从 O(N) 降至 O(1)

#### C. 可信时间戳协议（RFC 3161 简化版）

**理由**：签名绑定可信时间，防止时间篡改。

**实现方案**：
- 签名数据 = `event_data + timestamp + prev_hash`
- 时间戳来自外部可信源（或服务端时钟 + 签名）
- 验证时检查时间戳单调递增

### 6.4 优先级与预期收益

| 优化项 | 优先级 | 预期收益 |
|--------|--------|----------|
| 密钥派生与轮换 | **P0** | 满足安全合规要求 |

## 7. Storage 存储层

### 7.1 当前架构

- 5 种存储后端：Memory、PostgreSQL、Elasticsearch、ClickHouse、Redis、S3
- 统一 `StorageBackend` Protocol 接口
- 各后端独立实现，无中间层

### 7.2 当前问题/瓶颈

| 问题 | 严重度 | 描述 |
|------|--------|------|
| **无分层缓存** | 🔴 高 | 热数据（最近事件）和冷数据（历史归档）使用同一后端 |
| **无自动降级** | 🔴 高 | 后端不可用时直接异常，无 fallback 机制 |
| **无统一查询层** | 🟡 中 | 不同后端 API 不同，切换存储需改业务代码 |
| **缺少数据分区策略** | 🟡 中 | 大量 Agent 事件无自动分区/分片 |
| **Redis 无持久化保障** | 🟢 低 | Redis 作为缓存层，进程重启可能丢数据 |

### 7.3 推荐算法

#### A. 分层存储架构（Hot/Warm/Cold Tiering）

**理由**：不同数据温度使用不同存储，优化成本和性能。

**实现方案**：
```
Hot Tier (Redis)     → 最近 1 小时事件，实时查询，TTL 过期
Warm Tier (ClickHouse/PG) → 最近 30 天事件，聚合分析
Cold Tier (S3)       → 30 天以上事件，归档存储，压缩
```

- 自动迁移：后台任务按时间将 Hot → Warm → Cold
- 查询时自动路由：按时间范围选择最优后端
- 成本节约：S3 存储成本仅为 ClickHouse 的 1/10

#### B. 存储适配器链（Chain of Responsibility）

**理由**：实现自动降级和多后端写入。

**实现方案**：
```python
class ChainedStorage(StorageBackend):
    """写入时依次写入所有后端，读取时按优先级查询"""
    
    def __init__(self, backends: list[StorageBackend]):
        self._write_chain = backends  # 全部写入
        self._read_chain = list(reversed(backends))  # 最快的先查
    
    def save(self, agent_id, event):
        for backend in self._write_chain:
            try:
                backend.save(agent_id, event)
            except Exception:
                logger.warning(f"Backend {backend} write failed, continuing")
    
    def get_events(self, agent_id):
        for backend in self._read_chain:
            try:
                return backend.get_events(agent_id)
            except Exception:
                continue
        raise StorageError("All backends failed")
```

#### C. 一致性哈希分区

**理由**：大规模 Agent 事件的自动分片。

**实现方案**：
- 按 `agent_id` 做一致性哈希，分配到不同分片
- 支持动态扩缩容（虚拟节点重平衡）
- 每个分片独立的存储后端实例

|--------|--------|----------|
| 分层存储 | **P0** | 存储成本降低 70%，查询延迟降低 50% |
